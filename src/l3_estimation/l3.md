# newvision L3 状态估计层开发规划

## 1. 目标与边界

L3 的职责是将 L2 输出的二维装甲板检测，转换为 L4 可以直接预测和规划的目标车辆状态：

```text
ArmorDetection[] + 图像时间戳
  -> 单板 PnP
  -> camera / gimbal / world 坐标转换
  -> 装甲板 yaw 重投影优化
  -> 同车分组与装甲面关联
  -> 整车 EKF
  -> TargetState[]
```

L3 不负责：

- 神经网络推理、颜色过滤和 NMS（属于 L2）；
- 最终选择攻击哪辆车、选择哪块装甲板（属于 L4）；
- 弹道、延迟补偿和瞄准角计算（属于 L4）；
- 是否开火和串口命令生成（属于 L5）。

L3 对 Runtime 只暴露 `TargetEstimator`，其他几何和滤波细节不向外泄漏。

本项目按当前比赛规则只处理四装甲板车辆。所有 tracker 固定维护 0、1、2、3
四个物理装甲面，不再保留二装甲板、三装甲板或可变 `armor_count` 分支。

## 2. 冻结的数据契约

### 2.1 输入

```cpp
std::vector<L2Perception::ArmorDetection>
```

每个检测必须包含：

- 四角点，顺序固定为左上、右上、右下、左下；
- `class_id`，用于确定车辆编号和大小装甲板；
- 颜色；
- 检测置信度。

每次更新还必须提供该图像的曝光时间戳：

```cpp
L3Estimation::TimePoint timestamp
```

图像时刻的云台姿态通过构造时注入的 `GimbalPoseProvider` 查询，L3 不直接依赖串口类。

### 2.2 输出

```cpp
std::vector<L3Estimation::TargetState>
```

L3 同时维护所有可跟踪车辆；L4 从数组中选择最终目标。无有效目标时返回空数组，不使用 `robot_id == -1` 表示一个无效目标。

`TargetState` 的 11 维状态顺序固定为：

```text
x = [xc, vx, yc, vy, zc, vz, yaw, yaw_rate, r1, r2-r1, z2-z1]
```

字段映射：

```cpp
target.center        = {x[0], x[2], x[4]};
target.velocity      = {x[1], x[3], x[5]};
target.yaw           = x[6];
target.yaw_rate      = x[7];
target.radius        = x[8];
target.radius_offset = x[9];
target.height_offset = x[10];
target.covariance    = P;
target.timestamp     = filter_time;
```

协方差必须使用与状态相同的行列顺序，类型固定为 `11 x 11`，不能在各模块中重新排列。

## 3. 坐标、单位和角度约定

全部约定与 `/home/rm/rm_summmer/tongjiceshi` 一致。

### 3.1 相机坐标系

OpenCV 相机坐标系：

```text
x_camera：图像右方
y_camera：图像下方
z_camera：相机前方
```

### 3.2 云台和局部世界坐标系

```text
x：前
y：左
z：上
```

世界系由图像时刻的云台姿态提供方向，原点位于云台旋转中心。它是 IMU 稳定的局部世界系，不是赛场地图绝对坐标系。

坐标变换：

```cpp
p_gimbal = R_camera_to_gimbal * p_camera + t_camera_to_gimbal;
p_world  = R_gimbal_to_world * p_gimbal;
```

相机外参语义必须固定为“相机坐标中的点变换到云台坐标”，不得混用逆变换。

### 3.3 yaw

```text
yaw = 0：指向世界系 +x
yaw > 0：从 +x 转向 +y，即向左转
范围：(-pi, pi]
单位：rad
```

`TargetState::yaw` 表示 0 号装甲板局部 `+x` 法线方向。

第 `id` 个装甲面的方向和位置为（`id` 只能取 0、1、2、3）：

```cpp
angle = normalizeAngle(yaw + id * pi / 2.0);

use_second_group = (id == 1 || id == 3);
r = radius + (use_second_group ? radius_offset : 0.0);

armor_x = center.x() - r * std::cos(angle);
armor_y = center.y() - r * std::sin(angle);
armor_z = center.z() + (use_second_group ? height_offset : 0.0);
```

这里必须是减号。初始化车心时使用相反公式：

```cpp
center_x = armor_x + radius * std::cos(armor_yaw);
center_y = armor_y + radius * std::sin(armor_yaw);
center_z = armor_z;
```

### 3.4 单位

```text
位置、半径、外参平移：m
线速度：m/s
角度：rad
角速度：rad/s
时间差：s
重投影误差：pixel
```

## 4. 模块设计

### 4.1 `types.hpp`

职责：集中定义 L3 的稳定数据类型。

包含：

- `TimePoint`；
- `ArmorSize`；
- `ArmorPose`；
- `ArmorObservation`；
- `StateIndex`、`StateVector`、`StateCovariance`；
- `TargetState`。

该文件是 L3 与 L4 的唯一数据契约。修改字段或状态顺序时必须通知 L4 负责人。

### 4.2 `PnpSolver`

职责：

```text
ArmorDetection + ArmorSize -> ArmorPose(camera)
```

要求：

- 小装甲板尺寸：`0.135 x 0.055 m`；
- 大装甲板尺寸：`0.230 x 0.055 m`；
- 使用 `SOLVEPNP_IPPE`；
- 拒绝面积过小、共线、非有限值和 `z <= 0` 的结果；
- PnP 只负责相机系位姿，不做目标选择和滤波。

后续应使用 `solvePnPGeneric` 获取 IPPE 的多个候选解，并按正深度和重投影误差选择最优解。

### 4.3 `TargetEstimator`

L3 的唯一门面，负责每帧总编排：

1. 按时间戳查询云台姿态；
2. 对每块检测执行 PnP；
3. 完成 camera -> gimbal -> world；
4. 计算原始 yaw 并做重投影 yaw 优化；
5. 生成 `ArmorObservation`；
6. 按 `robot_id` 对观测分组；
7. 所有 tracker 只预测一次；
8. 进行同帧装甲面一对一关联；
9. 更新对应车辆 EKF；
10. 更新生命周期并删除过期 tracker；
11. 发布有效的 `TargetState[]`。

姿态缺失时不能使用当前串口姿态冒充图像时刻姿态。本帧观测应丢弃，但已有 tracker 可以按策略仅预测并降低可信度。

### 4.4 `EkfTracker`

一个 `EkfTracker` 只表示一辆车，内部包含：

- 11 维状态 `x`；
- `11 x 11` 协方差 `P`；
- 车辆编号；
- 当前关联的装甲面；
- 最后观测时间；
- `Detecting / Tracking / TemporaryLost / Lost` 生命周期。

第一版状态转移采用匀速模型：

```text
xc  += vx       * dt
yc  += vy       * dt
zc  += vz       * dt
yaw += yaw_rate * dt
```

`r1`、`r2-r1`、`z2-z1` 采用随机游走或常量模型。

过程噪声使用分段白噪声模型：位置/速度块和 yaw/yaw_rate 块随 `dt` 构造，不能把固定 Q 用在不同帧间隔上。

### 4.5 观测模型

对已关联到物理装甲面 `id` 的状态，先由整车状态计算装甲板位置：

```text
p_armor = h_position(x, id)
armor_yaw = normalize(yaw + id * pi / 2)
```

推荐与基准项目一致，使用如下观测：

```text
z = [bearing_yaw, bearing_pitch, distance, armor_yaw]
```

其中：

```cpp
bearing_yaw   = atan2(y, x);
bearing_pitch = atan2(z, std::hypot(x, y));
distance      = p.norm();
```

以下创新分量必须归一化角度：

- `bearing_yaw`；
- `bearing_pitch`；
- `armor_yaw`。

测量协方差 R 至少应根据下列因素动态调整：

- 距离；
- PnP 重投影误差；
- 装甲板正对程度；
- L2 检测置信度。

第一版可以先使用对角 R，但所有参数必须集中配置，不能散落为魔法数字。

### 4.6 装甲面关联

关联在同一车辆的一帧观测内整体完成，要求：

- 一块观测最多匹配一个物理装甲面；
- 一个物理装甲面最多接收一块观测；
- yaw 残差必须经过角度归一化；
- 超过位置、yaw 或马氏距离门限的匹配必须拒绝；
- 双板观测应满足相邻面相差 90 度的关系；
- 切换到 1/3 号面时正确使用第二组半径和高度。

装甲板最多四块，第一版直接枚举排列求最小总代价即可，不必引入复杂匹配库。

### 4.7 多车辆生命周期

`TargetEstimator` 使用：

```cpp
std::unordered_map<int, EkfTracker> trackers_;
```

按 `robot_id` 维护独立 tracker。建议状态机：

```text
Lost -> Detecting -> Tracking -> TemporaryLost -> Lost
```

规则：

- 新目标连续命中若干帧后才发布为 `Tracking`；
- 短时丢失只预测，不立即删除；
- 超过最大丢失时间后删除；
- 时间戳倒退、`dt` 过大、状态非有限或半径越界时重置；
- `TargetState.timestamp` 始终等于当前滤波状态对应时刻。

## 5. 实施阶段与验收标准

### 阶段 0：冻结接口和约定

任务：

- 固定 `TargetState` 和 11 维状态顺序；
- 将本文坐标、单位、yaw 和装甲面编号写入代码注释；
- 确认 `class_id -> robot_id / ArmorSize` 映射，装甲面数量恒为 4。

验收：

- L3 头文件可以独立编译；
- L3 与 L4 对字段和单位没有歧义；
- 不再出现 `VehicleState` 与 `TargetState` 两套输出类型。

### 阶段 1：完成单板几何链路

任务：

- 完善 PnP 输入校验和多解选择；
- 加载 `R_camera_to_gimbal`、`t_camera_to_gimbal`；
- 按图像时间获取 `R_gimbal_to_world`；
- 输出 `ArmorObservation`。

验收：

- 合成投影能恢复已知相机系位置；
- 零姿态下，相机前方点转换为世界系 `+x`；
- 相机图像右方对应世界系 `-y`；
- 相机图像上方对应世界系 `+z`；
- 缺失姿态、退化四点和相机后方解被安全拒绝。

### 阶段 2：完成 yaw 重投影优化

任务：

- 从 PnP 旋转矩阵计算原始世界 yaw；
- 按固定装甲板 pitch 模型重投影四角点；
- 搜索或一维优化 yaw；
- 保存原始 yaw、优化 yaw 和最终像素误差。

验收：

- 合成数据的 yaw 误差可控；
- 跨越 `-pi/pi` 时结果连续；
- 优化后的代价不高于原始 yaw；
- 如果不同车辆类别使用不同装甲板 pitch，则规则必须有明确配置和测试。

### 阶段 3：完成单车辆整车 EKF

任务：

- 实现 11 维初始化、预测和单板更新；
- 实现非线性观测函数和雅可比；
- 实现角度创新归一化；
- 输出固定维度协方差。

验收：

- 静止目标的速度和 yaw_rate 收敛到零附近；
- 匀速目标的位置和速度可稳定跟踪；
- 匀速旋转目标的 yaw/yaw_rate 连续；
- `P` 保持有限、对称且半正定；
- 时间戳异常不会产生负 dt 或状态爆炸。

### 阶段 4：装甲面关联和双板更新

任务：

- 建立固定四装甲面模型，面间角固定为 90 度；
- 完成同帧一对一关联；
- 支持一帧多观测顺序更新；
- 正确维护 r1/r2 和 z1/z2。

验收：

- 双板不会关联到同一物理面；
- 0/2 和 1/3 号面切换时半径、高度连续；
- 45 度附近噪声不会造成每帧反复跳面；
- 错误类别或离群观测不会强制更新滤波器。

### 阶段 5：多车辆管理与正式输出

任务：

- 按 `robot_id` 分组；
- 实现 tracker 创建、确认、暂失和淘汰；
- 每帧返回所有有效 `TargetState`；
- 记录创新、关联结果和拒绝原因。

验收：

- 多辆车同时出现时状态互不污染；
- 单车丢失不会删除其他车辆；
- 短暂遮挡后能继续跟踪；
- L3 不包含目标优先级和开火逻辑。

## 6. 测试规划

建议新增以下测试：

```text
tests/pnp_solver_smoke.cpp             已有，继续扩展多解和异常输入
tests/coordinate_transform_smoke.cpp   camera/gimbal/world 轴向和外参
tests/yaw_optimizer_smoke.cpp           合成角点恢复 yaw
tests/ekf_tracker_smoke.cpp             静止、匀速、匀速旋转
tests/armor_association_smoke.cpp       单板、双板、切板、角度跨 pi
tests/target_estimator_smoke.cpp        L2 检测到 TargetState 的端到端合成测试
```

测试优先使用合成数据，不依赖真实相机、串口或模型。真实视频用于回放验证，不替代确定性的单元测试。

每个测试至少检查：

- 数值是否有限；
- 单位和轴向是否正确；
- 时间戳是否正确传播；
- 角度是否归一化；
- 协方差维度、对称性和合理范围；
- 异常输入是否安全拒绝。
